# 回测框架需求文档

## 1. 项目概述

构建一个事件驱动的量化回测框架，初期支持 A 股日频回测，架构上可扩展至 ETF、期货等资产类型。框架提供当前仿真时间点，策略自由访问外部数据并生成信号（含权重）驱动仓位管理。

---

## 2. 资产支持

**2.1 当前必须支持**
- A 股股票：含分红、拆送股、停复牌、退市处理

**2.2 可扩展资产类型（预留扩展点，初期不实现）**

| 资产类型 | 扩展时需额外处理的特性 |
|---|---|
| ETF | 申赎机制、折溢价、成分股权重 |
| 期货 | 到期交割、换月移仓、保证金与强平 |

**2.3 扩展性要求**
- 资产类型通过抽象基类（`Asset`）统一接口，新增资产只需继承并实现对应的企业行动处理逻辑，无需改动核心引擎。

---

## 3. 事件系统

**3.1 日频事件序列**

每个交易日包含以下有序事件：

| 优先级 | 事件 | 说明 |
|---|---|---|
| 1 | `CorporateActionEvent` | 框架处理除权除息：送股增加持仓份额，现金红利入账，成本基础同步调整 |
| 2 | `DelistEvent` | 框架检测退市股票，强制以当日开盘价清仓 |
| 3 | `SignalEvent` | 策略基于 T 日数据计算信号 |
| 4 | `OrderEvent` | 根据信号生成 T+1 委托单，经规则链校验 |
| 5 | `FillEvent` | T+1 开盘成交回报，更新持仓与资金 |
| 6 | `PortfolioUpdateEvent` | 更新组合净值、风险指标 |

**3.2 关键约定**
- 策略在 T 日收盘后运行，只能看到 T 日及之前的数据。
- 委托在 T+1 开盘时以开盘价成交，遵守 T+1 规则（当日买入不可当日卖出）。
- 企业行动和退市处理先于策略信号，保证策略看到的状态是已调整后的。

---

## 4. 策略接口

**4.1 预热期**

策略声明所需的历史数据长度，框架在预热期内正常推进时间和数据，但不执行任何委托。预热结束后才开始计入绩效。

```python
class MyStrategy(Strategy):
    warmup_period = 250  # 需要 250 个交易日历史数据
```

预热期内 `on_bar` 正常调用，策略可初始化指标和模型状态，但框架忽略所有信号输出。

**4.2 数据访问**

框架向策略提供 `context.current_date`（T 日）。策略可访问任意外部数据源，无访问方式限制。

```python
def on_bar(context):
    dt = context.current_date  # 策略只应访问 <= dt 的数据

    df = duckdb.query(f"""
        WITH universe AS (
            SELECT symbol FROM cn_stock_bar1d
            WHERE date = '{dt}' AND volume > 1e8
        ),
        factor AS (
            SELECT symbol, pe_ttm FROM cn_stock_fundamental
            WHERE date <= '{dt}'
            QUALIFY ROW_NUMBER() OVER (PARTITION BY symbol ORDER BY date DESC) = 1
        )
        SELECT u.symbol, f.pe_ttm
        FROM universe u JOIN factor f USING (symbol)
        ORDER BY pe_ttm LIMIT 20
    """).df()
```

**使用须知：** 框架不拦截数据访问。所有查询必须以 `context.current_date` 为边界，不得访问该日期之后的数据。回测结果的正确性由策略作者负责。

**4.3 Universe 管理**

框架维护每日可交易标的列表，策略可通过 `context.tradeable` 获取当日可交易的全量标的。框架负责处理标的的进出（IPO 上市、退市摘牌、停复牌），策略无需自行过滤。

**4.4 信号输出**

策略输出目标权重信号列表：

```
Signal:
  symbol : 标的代码
  weight : 目标权重（相对组合净值；0 = 清仓，0.1 = 占组合 10%）
  meta   : 附加信息（可选）
```

- 返回空列表 `[]` 语义为维持现状，不触发任何交易。
- 如需清仓，策略应显式返回 `weight=0` 的信号。
- 初期仅支持多头（`weight >= 0`），做空预留扩展。

**4.5 策略生命周期**

| 回调 | 触发时机 |
|---|---|
| `on_start(context)` | 回测开始，初始化策略状态 |
| `on_bar(context)` | 每个交易日收盘后，计算并输出信号 |
| `on_adjustment(context, action)` | 企业行动发生时（可选，框架已自动处理，策略可额外响应）|
| `on_end(context)` | 回测结束，输出汇总信息 |

## 5. 执行模型

**5.1 成交价格**
- T 日收盘后生成信号，T+1 开盘价成交。
- 停牌股票跳过，无法成交。

**5.2 交易成本**

| 成本项 | 说明 |
|---|---|
| 佣金 | 按成交金额双边收取，费率可配置（默认万分之三），有最低佣金（默认 5 元）|
| 印花税 | 卖出单边，税率可配置（默认 0.1%）|
| 滑点 | 按成交金额比例估算，可配置（默认万分之五）|

**5.3 现金缓冲**

框架预留一定比例现金（默认 2%）不参与投资，防止手数取整后超额委托导致资金不足。缓冲比例可配置。

**5.4 调仓阈值**

目标权重与当前权重之差超过阈值才触发交易，避免信号微小变化产生无效换手。

```
仅当 |target_weight - current_weight| > rebalance_threshold 时才生成委托
```

阈值基于权重偏差（比例），默认关闭（0），可按需配置（如 0.01 = 1%）。

**5.5 委托校验规则链**

委托单在成交前经过以下规则校验，顺序固定：

| 规则 | 内容 |
|---|---|
| `HaltGuard` | 停牌或无行情时拒绝委托 |
| `LimitGuard` | 涨跌停板完全锁死时拒绝委托 |
| `T1Guard` | 买入委托标记为当日不可卖出 |
| `AvailableSharesGuard` | 卖出不超过可卖份额，按手数（100 股）取整；科创板 200 股起 |
| `CashGuard` | 资金不足时按可用资金缩减买入量，缩减后自动对齐手数 |
| `VolumeCapGuard` | 单笔委托不超过当日总成交量的 N%（默认关闭，ratio=1）|
| `DelistGuard` | 禁止买入已退市股票 |

**5.6 企业行动（框架自动处理）**
- 送股/转增：在除权日按持仓份额增加股票数量，成本基础不变（自动摊薄），新增份额当日不可卖出（T+1）。
- 现金分红：在派息日按持仓份额将税后金额入账，同步降低持仓成本基础。
- 历史价格数据不做复权调整，策略如需计算历史因子应使用外部复权数据。

**5.7 退市处理（框架自动处理）**
- 检测依据：独立的退市状态字段（不依赖股票名称匹配）。
- 检测时机：企业行动处理完毕后、策略信号生成前。同日有分红派息时，先派息再清仓。
- 处理方式：以当日开盘价强制全仓清仓，正常收取交易成本，记录到交易日志。
- 若当日无行情数据，以持仓最后已知价格清仓。

**5.8 其他约束**
- 初期不内置组合级约束（个股上限、行业敞口），由策略层自行实现。

## 6. 绩效分析

回测结束后框架自动输出标准绩效报告，相对基准计算。

**6.1 基准**
- 基准收益序列由用户以 DataFrame（含 date 列）形式注入，框架不负责加载。
- 所有相对指标（Alpha、Beta、信息比率）均基于此基准计算。

**6.2 必须输出的指标**

| 类别 | 指标 |
|---|---|
| 收益 | 累计收益、年化收益、基准年化收益、超额年化收益 |
| 风险 | 年化波动率、最大回撤、最大回撤持续时间 |
| 风险调整收益 | Sharpe 比率、Calmar 比率、信息比率（IR）|
| 相对基准 | Alpha、Beta |
| 交易统计 | 年化换手率、平均持仓数、胜率、盈亏比 |
| 时序分析 | 月度收益热力图、滚动 6 个月 / 12 个月 Sharpe |

**6.3 输出格式**
- 结构化数据（dict / DataFrame），便于程序化处理。
- 可选输出 HTML 报告，含图表。

## 7. 参数优化

框架提供内置参数优化接口，支持对策略参数进行系统化搜索。

- **网格搜索**：枚举参数空间的所有组合，对每组参数独立运行回测。
- **Walk-forward 滚动验证**：将回测区间分为训练集和测试集滚动推进，防止过拟合，支持多进程并行。
- 优化目标可配置（默认 Sharpe，也可指定其他指标）。
- 优化结果输出参数 → 绩效的映射表，供人工分析。

## 8. 非功能性需求

| 维度 | 要求 |
|---|---|
| 灵活性 | 策略可使用任意数据源和查询方式，框架不限制 |
| 可扩展性 | 资产类型、事件类型、执行模型均通过接口扩展，不修改引擎核心 |
| 可测试性 | 核心模块（事件队列、规则链、仓位计算、成本计算、绩效计算）可单独单测 |
| 可观测性 | 每个事件处理记录结构化日志，支持回放调试 |
| 技术栈 | Python，数据处理使用 polars / pyarrow / duckdb，不使用 pandas |
